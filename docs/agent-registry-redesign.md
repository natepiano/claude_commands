# Agent Registry Redesign — one place for family/agent/effort assignments

## Goal

Every external-CLI agent consumer (the `/plan:delegate` pipeline, the `~/.zshrc`
CLI aliases, clean-fix stages, `/ask_a_friend`) resolves "which CLI do I run,
with which model and effort" through one config file and one resolver. Switching
a major function between codex and claude is a single assignment edit; each
family carries a complete, always-valid description of how it runs every
sub-task, so nothing else needs touching on a switch.

This replaces today's per-consumer confs (`config/delegate.conf`,
`scripts/cli_agent/agent-assignment.conf`, the agent keys in
`scripts/clean-fix/agent-assignments.conf`) and the codex-only delegate
launchers (`codex_implement.sh` / `codex_review.sh`).

## Data model

Terminology: a **family** is a CLI vendor (`codex` | `claude`). An **agent** is
a specific model within a family (`gpt-5.6-sol`, `opus`, …). A **function** is a
major consumer (`delegate`, `cli`, `cleanfix`, `ask_a_friend`) containing
**sub-tasks** (`delegate.implementation`, `cli.commit_prep`, …). A task's full
name is `<function>.<subtask>` — exactly two segments.

Three layers:

1. **Function → family** (`[assignments]`): the only switch. `delegate=codex`
   routes all of `/plan:delegate` through the codex set.
2. **Function × family → sub-task table** (`[<function>.<family>]`): each
   function carries *both* sets, fully specified at all times. Each row maps a
   sub-task to an `agent:effort` pair (bare `agent` = omit the effort flag, use
   the CLI's default).
3. **Family → valid agents with valid efforts** (`[<family>.agents]`): the
   validation catalog. Codex entries are synced from
   `~/.codex/models_cache.json`, which provides per-model
   `supported_reasoning_levels` — effort validity is per-agent, not per-family.
   Claude entries are hand-maintained.

Resolution for task `T = f.s`:

```
family = [assignments] f          # exact task-name key f.s wins over f if present
pair   = [f.family] s             # "agent:effort" or "agent"
validate: agent ∈ [family.agents], effort ∈ that agent's effort list
→ (family, agent, effort)
```

The exact-task override (`delegate.review=claude` beating `delegate=codex`) is
supported by the resolver for one-off cross-vendor setups, but function-level
assignment is the norm and the only thing `/agent` writes.

## New `config/agents.conf`

The file keeps its path; the schema is replaced wholesale. Initial content
(claude-side leveling values are first-cut defaults, tune freely):

```ini

# agent:effort table each function uses per family.
#
# Resolver: scripts/agents/agents_config.sh (agents_resolve <function>.<subtask>).
# [codex.agents] is generated by scripts/agents/sync_codex_catalog.sh from
# ~/.codex/models_cache.json (per-model supported efforts). [claude.agents]
# is hand-maintained.

# ── major function → family: the switch ──
[assignments]
delegate=codex
cli=codex
cleanfix=codex
ask_a_friend=codex
team_review=codex
api_review=codex
module_review=codex

# ── delegate (/plan:delegate) ──

review=gpt-5.6-sol:high
mechanical=gpt-5.6-sol:medium

escalation=fable:max

reviewer=opus:max

gpt-5.6-terra=low,medium,high,xhigh
gpt-5.6-luna=low,medium,high,xhigh

[claude.agents]

opus=low,medium,high,xhigh,max
sonnet=low,medium,high,xhigh,max
```

Notes:

- Codex effort lists above are placeholders; the sync script writes the real
  per-model lists from `supported_reasoning_levels` in the cache (e.g. gpt-5.5
  reports low/medium/high/xhigh; a smaller model may report fewer).
- Claude agent names are the CLI's short aliases: fable, opus, sonnet.
- The old `[codex] model=` / `[claude] model=` defaults, `[*.models]`, and
  `[*.efforts]` sections are gone — the pair *is* the profile, and per-agent
  effort lists replace family-wide effort lists.

## Resolver — `scripts/agents/agents_config.sh`

Rewritten around the new schema. Public API:

- `agents_resolve <task>` — sets `AGENT_FAMILY`, `AGENT_MODEL`, `AGENT_EFFORT`
  (effort may be empty = omit the flag). Errors loudly (nonzero, stderr names
  the missing/invalid piece and the allowed values) when: the function has no
  `[assignments]` entry, the set section or sub-task row is missing, the agent
  isn't in `[<family>.agents]`, or the effort isn't in that agent's list.
- `agents_resolve_print <task>` — one-line `task=… family=… agent=… effort=…`
  for `--status`-style output and tests.
- `agents_list_assignments` — render the full resolution table (every
  `[assignments]` entry, the active set's rows, resolved pairs) for `/agent
  status`.
- `agents_set_assignment <function> <family>` — validate family has a
  `[<function>.<family>]` section whose every row resolves, then rewrite the
  `[assignments]` line in place (awk rewrite, same pattern as
  `cli_agent_set` today).
- Keep: `agents_config_trim`, the low-level ini reader, and the
  freshness-triggered catalog sync at source time (unchanged behavior).

Flag assembly stays in consumers/launcher, but provide two helpers so the
codex/claude flag vocabulary lives in one place:

- `agents_codex_args` → `-m "$AGENT_MODEL"` plus
  `-c model_reasoning_effort="$AGENT_EFFORT"` when effort is non-empty
- `agents_claude_args` → `--model "$AGENT_MODEL"` plus `--effort "$AGENT_EFFORT"`
  when effort is non-empty

## Shared launcher — `scripts/agents/agent_exec.sh`

One family-dispatch function used by delegate, ask_a_friend, and clean-fix's
style scripts:

```
agent_exec <task> <mode:write|readonly> <working_dir> <prompt_file> <output_file> <log_file>
```

- Resolves via `agents_resolve <task>`.
- codex write: `codex exec -m … [-c model_reasoning_effort=…] --ephemeral
  --full-auto -C <dir> -o <output> "$PROMPT" > <log> 2>&1`
- codex readonly: same with `--sandbox read-only` instead of `--full-auto`.
- claude write: `claude --print --dangerously-skip-permissions
  --settings '{"sandbox":{"enabled":false}}' --model … [--effort …] --
  "$PROMPT" > <output> 2> <log>` (claude prints the final message to stdout;
  there is no transcript log — stderr is the log).
- claude readonly: same but `--permission-mode plan` replaces
  `--dangerously-skip-permissions` (read-only analogue of codex's sandbox flag).
- Extra per-consumer flags (e.g. codex `--add-dir`, prompt preambles) pass
  through via an env hook or trailing args — keep the signature minimal and let
  clean-fix keep its codex-specific preamble where it is if wiring it through
  is awkward.

## Sync script — `scripts/agents/sync_codex_catalog.sh`

- Target section becomes `[codex.agents]`; each visible model is written as
  `slug=<comma-joined efforts>` where efforts come from
  `supported_reasoning_levels[].effort` (order preserved). A model with no
  levels array gets an empty list (only bare-agent pairs validate against it).
- The `[codex] model=` mirror of `~/.codex/config.toml` disappears with the
  section; the selected-model prepend logic now prepends into `[codex.agents]`
  (with its efforts if the cache has them, empty otherwise).
- `--check` semantics unchanged. `scripts/agents/test_sync_codex_catalog.sh`
  updated for the new output shape.
- Claude alias staleness check (user request 2026-07-12): the same sync run
  parses the quoted aliases from `claude --help`'s `--model` flag text (today:
  'fable', 'opus', 'sonnet') and warns when one is missing from
  `[claude.agents]`. Warn-only, never auto-add — the effort list for a new
  alias is a human call, and exclusions like haiku are deliberate. A
  help-text wording change degrades to a no-op (no aliases parsed → no
  warning), never a false edit.

## Consumer migrations

### /plan:delegate

- `codex_implement.sh` → `scripts/delegate/implement.sh`,
  `codex_review.sh` → `scripts/delegate/review.sh`. Signature keeps
  `<session_dir> [working_dir] [prompt_file] [task]` where the old profile arg
  becomes the sub-task (`implementation` | `mechanical` | `escalation`;
  review script defaults to `review`). Internally: prefix with `delegate.`,
  call `agent_exec`.
- Provenance files (`impl_agent`, `review_agent`) record
  `task=/family=/agent=/effort=`. Log files go agent-neutral:
  `impl_codex.log` → `impl_agent.log`, `review_codex.log` → `review_agent.log`.
- `scripts/delegate/delegate_config.sh` and
  `scripts/delegate/test_delegate_config.sh` are deleted (superseded by the
  resolver and its tests). `config/delegate.conf` is deleted.
- `commands/plan/delegate.md`: update the three launcher call sites (lines
  ~202, ~255, ~367), rename `IMPLEMENTATION_PROFILE`/`FIX_PROFILE` wording from
  "profile" to "task", update the log-filename references and the provenance
  note near the end. Line ~193's factual claim ("profile effort is configured
  in `~/.claude/config/delegate.conf`; the model in `agents.conf`") must be
  rewritten — delegate.conf no longer exists.

### cli aliases

- `scripts/cli_agent/cli_agent.sh`: drop `agent-assignment.conf` and the local
  load/set logic. Map invocation → task: no args → `cli.interactive`; first
  arg (skill name) → `cli.<skill>`; unknown skill errors with the known list.
  Resolve, then exec as today (codex keeps `-c service_tier="fast"`;
  non-interactive claude keeps the `-- "/$invocation"` form). `--status` prints
  the cli rows of `agents_list_assignments`; `--set` is removed (use `/agent`).
- `~/.zshrc` needs no changes — aliases already pass the skill name.
- `commands/cli_agent.md` is replaced by `commands/agent.md` (below).

### clean-fix

- `agent-assignments.conf` keeps only `enabled=` per stage; agent/model/effort
  keys removed.
- `agent_assignments.sh`: `cf_load_stage_assignment` reads `enabled=` locally
  and fills agent/model/effort from `agents_resolve cleanfix.<stage>`. Variable
  meaning shifts: `STYLE_AGENT` = family, `STYLE_AGENT_MODEL` = agent — the
  `case "$STYLE_AGENT" in claude|codex)` dispatch in `style-eval-all.sh`,
  `style-eval-review-all.sh`, and `style-fix-worktrees.sh` keeps working. Drop
  the `${STYLE_AGENT_EFFORT:-xhigh}` fallbacks — empty effort now means "omit
  the flag", same as everywhere else (mirror cli_agent.sh's empty-effort
  handling in the codex branches).
- `clean-fix.sh` (the pipeline driver) is a consumer too: it calls
  `cf_load_stage_assignment` for all three stages (~lines 225-230) and logs
  `$STYLE_EVAL_AGENT` etc. (~327/338/347) — those log lines now name the
  family, which is fine, but reword to `family/agent` so the resolved model is
  visible. Its report-render step (~line 370) is today a bare `claude --print`
  with no model/effort — it becomes `agent_exec cleanfix.report write …`
  (new `report` row in the `[cleanfix.*]` sets).
- `clean-fix-usage.sh`: `print_stage_json`/`print_stage_text` (~95, ~438-444)
  render agent/model/effort columns — update to family/agent/effort, and
  effort may legitimately be empty (CLI default), so keep the `<default>`
  placeholder. Help text (~44-45) for `/clean_fix agent …` updates to the new
  semantics below.
- `/clean_fix`'s configure-agents surface shrinks: the scoped
  `agent|model|effort` subcommands (`/clean_fix agent eval claude`,
  `/clean_fix eval model opus`, …) lose their backing (per-stage keys are
  gone). `/clean_fix agent` becomes a status view (family + resolved rows via
  `cf_print_agent_assignments`) that points at `/agent cleanfix <family>` for
  switching and `/agent cleanfix.<stage> <agent>[:<effort>]` for row edits.
  `commands/clean_fix.md` (~269-300) and `scripts/clean-fix/README.md` (~line
  18) rewrite accordingly.
- Codex-specific plumbing inside these scripts stays, but must be
  family-conditional, not assumed: the codex exec-marker transcript filtering
  (`style-eval-all.sh` ~110), the codex usage/weekly-limit detection
  (`style-fix-worktrees.sh` ~434), and `clean_fix_report_parse.py`'s
  usage-limit wording/reason codes ("codex hit its usage limit",
  `codex-usage-limit` ~1166/1194/1836) — generalize the wording to name the
  resolved family, or key the strings on it.

### review commands (/team_review, /api_review, /module_review)

Today all three launch in-session Agent-tool subagents (`subagent_type:
Explore`). They move to registry-resolved external CLI agents so a whole
review team switches family with one `[assignments]` row:

- `commands/team_review.md` `<LaunchExpertTeam>` (~86-117): "Launch 3-5 agents
  in parallel using the Agent tool" becomes 3-5 parallel backgrounded
  `agent_exec team_review.expert readonly …` invocations, one per dimension
  lens, each with its own prompt file and output file under a session dir.
- `commands/api_review.md`: the 5 parallel reviewers (~63) →
  `agent_exec api_review.reviewer readonly …`; the 2 adversarial
  stress-testers (~116) → `agent_exec api_review.adversary readonly …`.
- `commands/module_review.md`: pass 1 (~79) and pass 2 over-large-files
  (~192) agents → `agent_exec module_review.reviewer readonly …`; pass 3
  doc-vs-code validation (~226) → `agent_exec module_review.validation
  readonly …`.

Shared migration mechanics for all three commands:

- External CLI agents inherit no session context. Each prompt file must be
  self-contained: the verbatim charter preamble (agents still Read
  `~/rust/nate_style/review-charter.md` themselves — both CLIs can read
  files), the review topic/intent/posture, explicit file paths, the
  dimension/lens, and the finding schema.
- Reviewers run `readonly` mode (codex `--sandbox read-only`, claude
  `--permission-mode plan`) — same as delegate review.
- The command backgrounds all `agent_exec` calls in one turn and yields;
  task-notifications signal completion. Synthesis, deduplication, and the
  decision walk stay in-session as today.
- Session dirs live under the scratchpad (per-agent
  `prompt_N.md`/`findings_N.txt`/`agent_N.log` plus provenance files), same
  layout as delegate sessions.

## `/agent` skill — `commands/agent.md` (replaces `/cli_agent`)

Thin wrapper over a new `scripts/agents/agent_admin.sh`:

- `/agent` or `/agent status` — full table: each function, its family, and the
  active set's resolved rows.
- `/agent <function> <codex|claude>` — `agents_set_assignment`; refuses if any
  row of the target set fails validation, naming the bad row.
- `/agent <function>.<subtask> <agent>[:<effort>]` — edit one row of the
  *active* set for that function (validated against `[<family>.agents]`).
- Relay script stdout/stderr exactly; on error, stop (no guessed corrections) —
  same contract as today's `/cli_agent`.
